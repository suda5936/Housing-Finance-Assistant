# Phase 12 배포·시연·운영 준비

## 1. 완료 범위

Phase 12에서는 원격 서비스나 계정을 임의로 만들지 않고, 로컬 데모를 재현하고 배포 가능 여부를 정직하게 판정하는 운영 기반을 구현했다.

```text
환경 설정 검사
→ 자동 출시 평가
→ 합성 데모 3세트 재현
→ API 준비 상태 확인
→ Web 프로덕션 빌드
→ 시연 리허설
→ 스테이징 배포·복구 훈련
→ 수동 출시 승인
```

현재 상태는 **로컬 단일 프로세스 데모 준비 완료, 공개 배포 차단**이다. PostgreSQL 마이그레이션 파일은 있지만 애플리케이션의 PostgreSQL 저장소 어댑터가 아직 없으므로, 프로덕션인 것처럼 배포하지 않는다.

## 2. 구현 산출물

| 파일·기능 | 역할 |
|---|---|
| `GET /health` | 프로세스 생존 확인 |
| `GET /ready` | 저장소·정책·로컬 LLM을 포함한 준비·축소 상태 |
| `apps/api/homefit_api/operations.py` | 환경별 배포 차단·경고 검사 |
| `apps/api/homefit_api/demo_data/scenarios.json` | 비식별 합성 데모 3세트 |
| `apps/api/homefit_api/demo.py` | 외부 서비스 없는 데모 계산·추출·순위 재현 |
| `npm run ops:check` | 현재 환경의 안전 설정 검사 |
| `npm run eval:release` | Phase 11 골든 출시 평가 |
| `npm run demo:check` | 데모 3세트 결과 재현 |

CI, 클라우드 계정, 복잡한 배포 프레임워크는 추가하지 않았다.

## 3. 생존·준비 상태

### `/health`

프로세스가 응답하는지만 확인한다. 외부 서비스 상태나 정책 검토 여부로 실패하지 않는다.

### `/ready`

다음 구성요소를 노출하되 비밀번호·URL·세션 정보는 반환하지 않는다.

- 업로드 경로 상위 디렉터리 쓰기 가능 여부
- 정책 카탈로그 로드와 독립 검토 완료 개수
- 현재 데이터 저장소 유형
- 로컬 Qwen3-4B 상태와 수동 폴백 가능 여부

상태 의미:

| 상태 | 의미 | 트래픽 처리 |
|---|---|---|
| `ready` | 필수·선택 구성요소가 모두 준비됨 | 가능 |
| `degraded` | 수동 경로로 핵심 계산·비교 가능 | 데모 가능, 원인 표시 |
| `blocked` | 필수 저장·정책 구성요소 실패 | 503, 트래픽 중단 |

현재는 메모리 저장소, 정책 검토 대기 또는 Ollama 선택 장애 때문에 `degraded`가 정상적인 로컬 데모 상태다.

## 4. 환경 분리 원칙

비밀값이 담긴 환경별 파일을 Git에 추가하지 않는다. 동일한 설정 스키마에 실행 환경이 안전하게 주입한 환경변수를 사용한다.

| 항목 | Development | Staging | Production |
|---|---|---|---|
| `APP_ENV` | `development` | `staging` | `production` |
| 저장소 | `memory` 허용 | PostgreSQL 어댑터 필요 | PostgreSQL 어댑터 필수 |
| CORS | localhost | 스테이징 Web origin | 실제 Web origin |
| Trusted Host | localhost | 스테이징 도메인 | 실제 도메인 |
| HSTS | `false` | HTTPS면 `true` | `true` 필수 |
| Ollama | loopback | 내부·로컬만 | 개인정보 검토 전 외부 전송 금지 |
| DB 비밀번호 | 로컬 예시 허용 | 비밀 저장소 | 비밀 저장소·교체 필수 |
| 정책 상태 | `official_check` | 검토 전 동일 | 승인 증거 없으면 동일 |

`npm run ops:check`는 현재 development 설정에서는 경고와 함께 성공한다. production 설정에서는 다음 항목을 차단한다.

- HSTS 미사용
- PostgreSQL 저장소 어댑터 미구현
- 기본 로컬 DB 비밀번호
- 와일드카드 CORS·Host
- 원격 Ollama
- 자동 출시 평가 실패

## 5. 합성 데모 데이터 3세트

모든 주소·인물·문서는 합성이며 네트워크나 Ollama 없이 동일 결과를 낸다.

| ID | 시연 목적 | 기대 1위 | 핵심 장면 |
|---|---|---|---|
| `balanced-seoul-youth` | 비용·통근·면적 균형 | A 망원동 안심집 | 모든 기여도 비교 |
| `cost-first-tight-budget` | 월 예산이 빠듯한 사용자 | A 신림 절약집 | 비용 가중치 60점 경계 |
| `commute-first-offline-fallback` | 통근 우선·외부 AI 장애 | B 종로 가까운집 | Qwen 없이 수동 입력 |

각 세트는 프로필, 후보 2개, 가중치, 합성 문서 텍스트와 기대 우승자를 포함한다. 실행 시 다음을 검증한다.

- 후보별 월평균·계약 총비용 계산
- 계산 버전·입력 해시를 포함한 순위
- 합성 문서 핵심 필드 6개 이상 추출
- 문서 인젝션·모순 없음
- 기대 1위와 실제 1위 일치
- 외부 서비스 호출 0회

```powershell
npm run demo:check
```

## 7. 배포 승인 순서

공개 배포 전에 다음 순서를 지킨다.

1. 커밋 SHA와 Web/API 버전을 기록한다.
2. 정책 문서 버전, 원문 SHA-256, 작성자·독립 검토자를 승인한다.
3. `npm run eval:release`가 자동 차단 0건인지 확인한다.
4. 수동 모바일·정책 검토 증거를 등록해 `release_ready=true`를 확인한다.
5. PostgreSQL 저장소 어댑터와 마이그레이션 실행기를 구현한다.
6. 스테이징에 실제 배포하고 `/health`, `/ready`, 전체 데모를 확인한다.
7. 백업을 만들고 복원 리허설을 수행한다.
8. TLS, 저장 암호화, 악성코드 검사, 중앙 로그·지표·알림을 확인한다.
9. 배포 책임자와 검토자가 함께 승인한다.

현재 4~8번의 실제 증거가 없으므로 공개 배포 승인을 내릴 수 없다.

## 8. DB 마이그레이션·백업·복구

`apps/api/migrations/001~006`은 신규 로컬 PostgreSQL 볼륨 초기화용이다. 현재 앱은 해당 DB에 읽고 쓰지 않는다.

Docker Compose 문법은 `docker-compose config`로 확인했다. Codex 환경에서는 Docker 사용자 설정 파일 접근 경고가 있었지만 Compose 모델 렌더링은 성공했다.

PostgreSQL 어댑터 구현 후 스테이징에서 사용할 절차:

```powershell
docker-compose up -d postgres
docker-compose exec -T postgres pg_isready -U homefit -d homefit
docker-compose exec -T postgres pg_dump -U homefit -d homefit -Fc -f /tmp/homefit.dump
```

복원은 기존 데이터를 덮을 수 있으므로 별도의 빈 스테이징 DB에서만 먼저 수행한다. 운영 DB 복원은 사용자 승인과 백업 확인 없이 실행하지 않는다.

마이그레이션 원칙:

- 적용 전 스키마·데이터 백업
- 한 번에 한 버전 적용
- 파괴적 변경은 expand → migrate → contract로 분리
- 애플리케이션 롤백 기간에는 이전 스키마 호환 유지
- 실패 시 새 트래픽 차단, 원인 기록, 이전 앱 버전 복귀

## 9. 코드 배포·롤백 절차

### 배포

1. 작업 브랜치 테스트·평가 통과
2. 팀원 리뷰 후 main 병합
3. 배포 대상 커밋 SHA 기록
4. 프로덕션 빌드 생성
5. 스테이징 배포 후 준비 상태와 데모 확인
6. 승인 후 운영 배포

### 롤백 조건

- `/ready`가 `blocked` 또는 503
- 계산 골든 불일치
- 정책 적격·부적격 경계 반전
- 오류율 5% 초과가 5분 지속
- 세션 삭제·마스킹 실패
- 출처 없는 최종 주장 노출

### 롤백 순서

1. 신규 트래픽 중단
2. 요청 ID와 최초 오류 시각 기록
3. 이전 검증 커밋으로 애플리케이션 복귀
4. 데이터 호환 확인
5. `/health`, `/ready`, `eval:release`, `demo:check` 재실행
6. 장애 보고서와 재발 방지 작업 생성

## 10. 장애 대응 역할

두 명 팀 기준으로 역할을 단순화한다.

| 역할 | 담당 | 행동 |
|---|---|---|
| Incident Lead | 현재 시연 조작자 | 시연 중단·수동 폴백 결정, 시간·요청 ID 기록 |
| Technical Backup | 다른 팀원 | API·Ollama·로그 확인, 이전 데모 경로 준비 |

우선 복구 순서:

1. Web 새로고침과 `/health` 확인
2. `/ready`의 차단 구성요소 확인
3. Ollama만 실패하면 문서 없이 수동 입력 계속
4. OCR만 실패하면 후보 값을 직접 입력
5. API 실패면 서버 한 번 재시작
6. 계산·정책 검증 실패면 결과를 보여주지 않고 준비된 오프라인 `demo:check` 결과로 구조 설명

## 11. 관측·알림 기준

현재 `/system/metrics`는 개발용 프로세스 내 지표이며 실제 알림 채널은 연결되지 않았다. 스테이징에서 다음 기준을 구성한다.

| 지표 | 경고 | 긴급 |
|---|---:|---:|
| HTTP 오류율 | 5분간 2% | 5분간 5% |
| API 평균 지연 | 5분간 1초 | 5분간 2초 |
| `/ready` | degraded 10분 | blocked 즉시 |
| 검증 실패율 | 5% | 10% |
| 삭제 실패 | 1건 | 즉시 |
| 디스크 사용률 | 75% | 85% |

오류 알림에는 요청 본문이나 토큰 대신 요청 ID, 템플릿 라우트, 상태 코드와 시간만 포함한다.

## 12. 3분 시연 시나리오

목표: 문제·차별점·결과를 빠르게 전달한다.

1. **0:00~0:25** — “월세만 싼 집이 아니라 내 삶의 총비용을 비교한다.”

4. **1:20~2:10** — 의사결정 카드의 월평균·총비용·부담률·기여도 제시
5. **2:10~2:35** — 비용 우선 또는 통근 우선으로 변경해 결과 변화 설명
6. **2:35~3:00** — 공식 출처, `official_check`, 데이터 삭제로 마무리

## 15. 심사위원 예상 질문

### 왜 4B 모델인가?

무료 로컬 실행과 개인정보 최소화를 위해 Qwen3-4B를 선택했다. 작은 모델이 계산·정책 판정을 맡지 않고, 코드 기반 계산기·규칙 엔진·검색·검증 게이트가 결정한다.

### 단순 계산기와 무엇이 다른가?

문서값 확인, 누락 질문, 정책 근거, 보증금 금융비용, 통근, 사용자 선호, 민감도와 계약 체크리스트를 하나의 상태 흐름으로 연결한다. 모든 숫자와 순위 기여도를 다시 추적할 수 있다.

### 추천을 믿어도 되는가?

계약·대출 승인을 보장하지 않는다. 정책 원문 검토가 끝나지 않으면 추천 카드도 `공식 확인 필요`로 종료하며, 계산 가정·버전·출처·누락값을 함께 보여준다.

### 환각은 어떻게 막는가?

LLM은 금액·자격·순위·도구 선택을 수정할 권한이 없다. 정책은 공식 도메인·해시·적용기간·독립 검토를 통과해야 하고, 최종 카드 전 provenance 검증을 수행한다.

### 개인정보는 어디에 저장되는가?

현재 데모는 단일 프로세스 메모리와 로컬 파일만 사용한다. 익명 토큰은 해시 저장되고 문서는 기본 24시간 또는 즉시 삭제된다. 원격 Ollama 주소는 차단한다.

### 실제 서비스로 바로 배포 가능한가?

아니다. PostgreSQL 어댑터, TLS·저장 암호화, 악성코드 검사, 중앙 관측·알림과 수동 검토 증거가 남아 있다. 현재 범위는 검증 가능한 로컬 MVP다.

## 16. 사용자 안내·개인정보 요약

- 이 서비스는 주거 선택을 돕는 비교 도구이며 법률·금융·대출·계약 결과를 보장하지 않는다.
- 주민번호, 계좌번호와 실제 이름을 입력하지 않는다.
- OCR 추출값은 사용자가 확인하기 전 계산에 사용하지 않는다.
- 정책·금리·계약 안전성은 공식기관·은행·등기·전문가에게 다시 확인한다.
- 결과 JSON을 내려받을 수 있고 세션과 데이터를 즉시 삭제할 수 있다.
- 로컬 데모 데이터는 모두 합성이며 실제 인물을 나타내지 않는다.
