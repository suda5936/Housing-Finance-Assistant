# Phase 4 정책 데이터·규칙 엔진

작성·확인 기준일: 2026-07-24  
구현 상태: 기술 구현 완료, 실제 정책 활성화는 사람의 2차 검토 대기

## 1. 목표와 안전 원칙

핵심 정책 3개의 조건을 버전이 고정된 데이터로 표현하고, 같은 입력에 항상 같은 사전자격 상태와 근거를 반환한다. 정책 판정에는 Ollama와 Qwen3-4B를 호출하지 않는다.

이 결과는 정책 수혜나 대출 승인을 보장하지 않는다. 서울시 지원은 자격심사 후에도 전산 추첨이 있으며, 대출 상품은 자산·신용·보증·대상주택에 대한 기금 및 수탁은행 심사가 남는다.

## 3. 정책 활성화 게이트

정책은 다음 조건을 모두 만족해야 `is_activatable=true`가 된다.

1. 공식 원문 파일의 SHA-256이 64자리 소문자 16진수로 등록되어 있다.
2. 규칙 상태가 `approved`다.
3. 작성자와 다른 검토자가 등록되어 있다.
4. 검토 시각이 등록되어 있다.
5. 문서·규칙 버전과 시행기간이 유효하다.

하나라도 빠지면 조건 평가를 수행하지 않고 `OFFICIAL_CHECK_NEEDED`와 `POLICY_REVIEW_PENDING`을 반환한다. 현재 저장된 3개 정책은 원문 파일 해시와 팀원 검토가 없으므로 의도적으로 `draft`다. 작성자나 검토자를 임의로 꾸며 활성화하지 않는다.

## 4. 상태 모델

| 상태 | 의미 |
| --- | --- |
| `ELIGIBLE` | 입력 기준상 기본요건 충족 가능. 선정·승인 보장 아님 |
| `INELIGIBLE` | 공식 문서로 표현 가능한 명시 조건 미충족 |
| `MISSING_INFORMATION` | 필수 입력이 없어 판정 보류 |
| `OFFICIAL_CHECK_NEEDED` | 검토 미완료, 예외, 자산·신용·은행 심사 등 공식 확인 필요 |
| `EXPIRED` | 해당 정책 버전이나 신청기간 종료 |

누락값은 `false`나 0으로 바꾸지 않는다. 조건 하나라도 필요한 값이 없으면 `MISSING_INFORMATION`과 누락 필드 목록을 반환한다.

## 5. 입력과 파생값

`EligibilityInput`은 기준일, 연령, 지역, 가구원 수, 세전 가구 연소득, 중위소득 비율, 순자산, 무주택·세대주 여부, 보증금, 월세, 면적, 계약금 납부비율, 중복수혜·중복대출 여부와 신청자 유형을 받는다.

정책 메타데이터에는 적용 지역·주거유형, 지원 또는 대출의 최대 금액·월 한도·기간·금리 설명, 중복수혜 제한, 예외, 필요한 서류와 공식 확인 항목을 각각 독립 필드로 저장한다. 화면과 후속 에이전트가 설명문을 다시 해석하지 않고 구조화된 값을 사용할 수 있다.

서울시 월세 환산 조건은 다음 파생값으로 계산한다.

```text
서울시 환산 월 주거비 = 월세 + 보증금 × 4.5% ÷ 12
```

숫자 비교는 `Decimal`을 사용한다. 입력 전체를 정렬된 JSON으로 직렬화한 SHA-256도 결과에 포함하므로 같은 입력과 버전을 재현할 수 있다.

## 6. API

### 정책 목록

```http
GET /policies
```

정책 코드, 운영기관, 문서·규칙 버전, 검토 상태, 활성화 가능 여부와 공식 출처를 반환한다.

### 정책 JSON Schema

```http
GET /policies/schema
```

카탈로그 작성 시 사용하는 Pydantic 기반 JSON Schema를 반환한다.

### 사전자격 판정

```http
POST /policies/{policy_code}/eligibility
```

결과에는 상태, 사유 코드, 누락 필드, 조건별 실제값·기대값·통과 여부, 원문 조항, 문서·규칙 버전, 출처 URL, 확인일, 원문 해시, 입력 해시와 공식 확인 항목이 포함된다.

- 규칙 평가기: `apps/api/homefit_api/policies.py`
- API: `apps/api/homefit_api/policy_api.py`

DB에는 정책 문서와 규칙 버전을 덮어쓰지 않고 별도 행으로 보존한다. 판정 결과는 사용한 문서·규칙 ID, 입력 해시, 상태, 사유와 조건별 결과를 함께 참조한다.

## 8. 갱신·승인·롤백 절차

1. 공식 사이트에서 새 원문을 내려받는다.
2. 파일 SHA-256, 공고일·시행일·종료일·확인일과 URL을 새 문서 버전에 기록한다.
3. 기존 규칙을 덮어쓰지 않고 새 `rule_version`을 만든다.
4. `changed_policy_codes(previous, current)`로 재검토와 재판정이 필요한 정책을 찾는다.
5. 작성자가 골든·경계 테스트를 갱신한다.
6. 다른 팀원이 원문 조항, 경계값, 예외, 서류와 체크리스트를 대조한다.
7. 검토자와 검토 시각을 기록하고 승인한다.
8. 문제가 생기면 새 버전을 `retired`로 바꾸고 직전 승인 버전을 활성화한다. 기존 판정 기록은 삭제하지 않는다.

## 9. 계약 전 체크리스트

체크리스트도 정책 문서와 규칙 버전에 연결된다.

- 서울시 지원: 주민등록·실거주·계약서 주소 일치, 중복수혜, 제출서류, 전산 추첨 확인
- 청년전용 보증부월세대출: 신청시기, 계약금 납부, 자산·신용·보증 심사, 은행 승인 확인
- 주거안정월세대출: 우대형 증빙, 대상주택, 자산·신용·보증 심사, 은행 승인 확인

## 10. 테스트 범위

최종 검증 결과 Phase 4 테스트 15개를 포함한 백엔드 테스트 46개와 프런트 테스트 2개가 모두 통과했다. 전체 백엔드 코드 커버리지는 93%이며 Ruff와 mypy 검사도 통과했다.

- 서울시 연령·지역·보증금·중위소득 경계값
- 보증금 월세 환산식
- 누락정보와 부적격 분리
