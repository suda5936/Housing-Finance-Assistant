# Phase 8 에이전트 오케스트레이션

## 1. 완료 범위

Phase 8에서는 Phase 2~7의 기능을 하나의 명시적 상태 머신으로 연결했다.

```text
동의 확인
→ 프로필 확인
→ 후보 2개 이상 확인
→ 선택 문서 추출
→ 사용자 문서값 확정
→ 필수 입력 점검
→ 정책 판정과 근거 연결
→ 후보별 비용 계산
→ 다기준 순위 계산
→ 최종 검증 게이트
→ 결정 카드 또는 공식 확인 필요
```

자유로운 LLM 대화 루프가 다음 행동을 결정하지 않는다. 상태와 전이는 고정 코드가 결정하며 계산, 정책 판정, 검색, 순위 산출은 기존의 검증된 결정론적 함수만 호출한다.

## 2. 상태 모델

| 상태 | 역할 | 자동 진행 여부 |
|---|---|---|
| `consent` | 세션 동의 확인 | 자동 |
| `profile` | 사용자 기본정보 요청 | 사용자 입력 대기 |
| `candidate_input` | 비교 후보 2개 이상 요청 | 사용자 입력 대기 |
| `extraction` | 선택 문서 추출 | 자동 |
| `user_confirmation` | OCR 필드 확인·수정 | 사용자 입력 대기 |
| `missing_info_check` | 필수 입력 점검 | 자동 |
| `clarification` | 가장 우선적인 누락·오류 질문 하나 제시 | 사용자 입력 대기 |
| `eligibility` | 정책 규칙과 근거 평가 | 자동 |
| `cost_calculation` | 후보별 비용 시나리오 계산 | 자동 |
| `ranking` | 사용자 가중치 기반 후보 비교 | 자동 |
| `verification` | 최종 안전 게이트 실행 | 자동 |
| `decision_card` | 모든 검증을 통과한 비교 카드 | 종료 |
| `official_check` | 공식 검토가 필요한 비교 카드 | 종료 |
| `failed` | 실행 한도 또는 복구 불가 오류 | 종료 |

사용자 입력 대기 상태에서는 자동 루프가 멈춘다. 세션과 실행 ID로 다시 조회하면 같은 상태, 질문, 중간 결과와 감사 로그를 그대로 받을 수 있다.

## 3. 추가 질문 우선순위

여러 값이 동시에 부족해도 한 번에 모든 질문을 던지지 않는다. 다음 순서로 가장 중요한 질문 하나만 반환한다.

1. 프로필
2. 비교 후보 2개
3. 업로드 문서와 후보 연결
4. 문서 필수 필드 확정
5. 확인할 정책
6. 정책 자격 입력
7. 후보별 비용 입력
8. 위험 근거와 비교 가중치

누락값은 0이나 `false`로 바꾸지 않는다. 금리, 초기비용, 위험점수, 선호 가중치는 에이전트가 임의로 채우지 않는다.

## 4. 사용자 수정값 우선

문서값은 다음 출처를 가진다.

- `document_confirmed`: 사용자가 확인한 OCR 추출값
- `user`: 사용자가 직접 수정·입력한 값

이미 `user` 출처인 값에 나중에 OCR 값이 들어와도 덮어쓰지 않는다. 문서가 연결된 후보는 보증금, 월세, 관리비, 면적의 최종 확인값이 비용·순위 입력과 일치해야 검증 게이트를 통과한다.

따라서 LLM 또는 OCR이 생성한 값이 사용자의 수정값을 다시 덮거나, 비용 계산에는 수정 전 값이 들어가는 경로를 차단했다.

## 5. 도구 레지스트리와 실행 순서

오케스트레이터가 호출할 수 있는 도구는 코드에 고정되어 있다.

| 도구 | 입력 | 출력 | LLM 사용 |
|---|---|---|---|
| `document.extract` | 세션·문서 ID | 구조화 문서 분석 | 없음 |
| `policy.eligibility_with_evidence` | 정책 코드·자격 입력 | 규칙 결과·조건별 인용 | 없음 |
| `cost.calculate` | `CostCalculationInput` | 3개 비용 시나리오 | 없음 |
| `ranking.compare` | `RankingRequest` | 순위·기여도·민감도 | 없음 |

Pydantic 입력·출력 모델을 통과하지 못한 값은 도구에 전달되지 않는다. Qwen3-4B는 현재 Phase 8 핵심 루프에서 호출하지 않으며 `llm_calls=0`, `llm_tokens=0`, `estimated_cost_krw=0`으로 기록된다. 이후 UI 문장 표현에 사용하더라도 결정론적 결과를 수정할 권한은 주지 않는다.

## 6. 중복 호출 캐시

도구 이름과 정규화 입력의 SHA-256을 캐시 키로 사용한다.

```text
cache key = tool_name + SHA-256(validated input)
```

같은 시나리오를 다시 실행하면 정책, 비용, 순위 결과를 재사용하고 `cached` 도구 호출로 감사 로그에 남긴다. 다른 입력은 다른 해시를 가지므로 이전 결과를 잘못 재사용하지 않는다. 캐시는 세션 삭제 또는 만료 시 AgentRun과 함께 제거한다.

## 7. 실행 한도와 실패 정책

기본 한도는 다음과 같다.

- 최대 상태 진행: 20회
- 최대 도구 호출: 20회
- 도구별 재시도: 1회
- 도구 제한시간: 30초
- LLM 호출: 0회
- LLM 토큰: 0개

API에서 한도를 조정할 수 있지만 서버 스키마의 상한을 넘을 수 없다.

| 실패 | 처리 |
|---|---|
| OCR·PDF 추출 실패 | 수동 후보 입력 요청 |
| 문서 필드 미확정 | 해당 필드 확인 요청 |
| 정책 도구 실패 | `official_check` |
| 비용 정보 부족 | `clarification` |
| 순위가 부분·비교 불가 | `clarification` |
| 비용 해시·순위값 불일치 | 최종 카드 차단 |
| 최대 진행·호출 초과 | `failed`, 추천 없음 |
| 출처 또는 규칙 검토 미완료 | `official_check` |

실패한 결과에는 결정 카드를 만들지 않는다. 정책 확인이 필요한 경우에는 비용·순위 결과를 보존하되 확정 추천이 아니라 공식 확인 필요 카드로 명확히 구분한다.

## 8. 최종 검증 게이트

최종 상태 전에 다음 게이트를 모두 실행한다.

1. 문서값이 사용자 확인 또는 수동 입력인지
2. 사용자 수정값이 비용·순위 입력에 반영됐는지
3. 정책 규칙 버전과 독립 검토된 인용이 있는지
4. 모든 비용 결과가 계산 완료이고 KRW 단위인지
5. 순위 입력의 비용 버전·입력 해시·실제 금액이 계산 결과와 일치하는지
6. 총점이 공개된 기준별 기여도의 합과 일치하는지
7. 계약 전 체크리스트가 승인 버전과 공식 근거를 가지는지
8. 보장하거나 법률 판단으로 오인될 표현이 없는지

사용자 보완이 가능한 실패는 `clarification`, 공식기관·독립 검토가 필요한 실패는 `official_check`로 보낸다.

## 9. 계약 전 체크리스트

LLM이 체크리스트 항목을 새로 만들지 않는다. Phase 4 정책 카탈로그의 버전형 체크리스트만 선택하고 Phase 7 출처 인용을 연결한다.

각 항목은 다음을 포함한다.

- 항목 코드와 규칙 버전
- 적용 정책
- 확인 행동
- 확인 주체
- 공식 출처 인용
- 규칙·출처 검토 상태
- 법률 판단이나 안전 보장이 아니라는 안내

규칙 작성자와 검토자가 같거나, 인용 출처가 `pending` 또는 `manual_snapshot`이면 체크리스트 검증 게이트를 통과하지 않는다.

## 10. 감사 로그와 개인정보 삭제

모든 실행에는 다음이 기록된다.

- 상태 전이 순서와 이유 코드
- 도구 이름과 입력·출력 SHA-256
- 완료·캐시·실패·수동 전환 상태
- 시도 횟수와 실행시간
- 검증 게이트 결과
- 상태 진행·도구 호출·재시도·캐시 사용량
- 오케스트레이션 버전과 시나리오 revision

원문 개인정보를 도구 감사 로그에 복제하지 않고 해시만 저장한다. 세션 삭제·24시간 만료 시 원본 문서, 문서 분석 메모리, AgentRun, 도구 캐시를 함께 제거하도록 Phase 2 삭제 흐름을 확장했다.

## 11. API

```http
POST /sessions/{session_id}/agent-runs
GET  /sessions/{session_id}/agent-runs/{run_id}
PUT  /sessions/{session_id}/agent-runs/{run_id}/context
POST /sessions/{session_id}/agent-runs/{run_id}/advance
```

모든 경로는 `X-Session-Token`을 요구한다.

`context` 갱신으로 정책 입력, 비용 입력, 순위 입력과 사용자 수정값을 추가한다. `scenario_update=true`이면 이전 결과를 비우고 시나리오 revision을 올린 뒤 비용·정책·순위를 다시 검증한다.

## 15. 최종 누락 검토

핵심·필수 기능을 다시 검토하여 다음 누락을 추가로 보완했다.

1. 순위 요청이 비용 해시만 복사하고 실제 금액을 바꾸는 경우까지 차단했다.
2. 순위의 정책 상태와 규칙 버전이 실제 자격 판정 결과와 일치하는지 검사한다.
3. 문서 확인값이 연결 후보의 비용·순위 입력과 일치하는지 검사한다.
4. 출처 상태만 임의로 `approved`로 바꾸지 못하도록 독립 검토자를 요구한다.
5. 세션 삭제 시 문서 분석 캐시와 AgentRun 메모리도 함께 제거한다.

현재 남은 운영 제약은 Phase 7의 정책·출처 교차검토다. 검토 전에도 상태 흐름, 비용 계산과 후보 비교를 실행할 수 있지만 결과는 의도적으로 `official_check`에서 종료한다.
