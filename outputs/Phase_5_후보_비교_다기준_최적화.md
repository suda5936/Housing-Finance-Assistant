# Phase 5 후보 비교·다기준 최적화

작성 기준일: 2026-07-24  
구현 상태: 완료

## 1. 목표

후보 주택 2~10개를 같은 입력과 가중치로 비교하여 재현 가능한 순위, 기준별 기여도, 후보 간 트레이드오프와 민감도를 반환한다. 계산과 순위에는 Qwen3-4B를 사용하지 않는다.

순위는 의사결정을 보조하는 상대 비교 결과다. 매물의 법적 안전성, 정책 수혜, 대출 승인 또는 계약 적합성을 보장하지 않는다.

## 2. 하드 제약과 선호 기준 분리

하드 제약은 점수로 상쇄할 수 없는 조건이다.

- 월 실질 주거비 상한
- 편도 통근시간 상한
- 보증금 상한
- 최소 전용면적
- 필수 자치구 목록

하드 제약을 넘으면 `hard_constraint_failed`로 분리하며 점수 순위에 포함하지 않는다. 하드 제약 확인에 필요한 값이 없으면 실패로 간주하지 않고 `not_comparable`로 보류한다.

선호 기준과 기본 가중치는 다음과 같다.

| 기준 | 방향 | 기본 가중치 |
| --- | --- | ---: |
| 월 실질 주거비 | 낮을수록 좋음 | 40% |
| 편도 통근시간 | 낮을수록 좋음 | 25% |
| 보증금 | 낮을수록 좋음 | 15% |
| 전용면적 | 높을수록 좋음 | 10% |
| 확인 필요 위험점수 | 낮을수록 좋음 | 10% |
| 생활 인프라 | 높을수록 좋음 | 0% |

가중치 합은 정확히 100이어야 하며 기준 하나의 최대 비중은 60%다. 60%를 넘는 강한 필수 선호는 점수 대신 하드 제약으로 표현한다.

## 3. 입력 출처와 비교 기준

### 비용

`monthly_effective_cost`에는 Phase 3 계산 엔진 결과를 사용하고 `cost_calculation_version`과 `cost_input_sha256`을 함께 입력한다. 모든 후보는 동일한 `base`, `optimistic` 또는 `conservative` 시나리오를 사용해야 한다. 서로 다른 시나리오의 후보는 요청 단계에서 거부한다.

기본 비교는 `base` 시나리오다. 미확정 정책지원금은 기본 비용에서 차감하지 않으며, 지원 가정 비교가 필요하면 모든 후보를 동일한 별도 시나리오로 다시 계산한다.

### 통근

편도 통근시간과 조회·입력 기준시각을 함께 받는다. 후보 간 기준시각 차이가 30일을 초과하면 `COMMUTE_REFERENCE_TIME_MISMATCH_OVER_30_DAYS` 경고가 발생한다.

### 위험점수

0은 확인된 위험신호 없음, 100은 확인 필요 신호가 매우 많음을 뜻한다. 점수와 함께 `risk_basis`를 필수로 받는다. Phase 6 전에는 사용자가 확인한 위험신호를 명시적인 산식으로 환산하고, 자동 법률판단으로 취급하지 않는다.

### 생활 인프라

기본 가중치는 0이므로 신뢰 가능한 자료가 없을 때 순위에 영향을 주지 않는다. 가중치를 부여하려면 0~100 점수와 다음 근거를 모두 입력해야 한다.

- `source`: 사용자 평가 또는 공개데이터
- `methodology`: 점수 산식
- `reference_at`: 평가 기준시각
- `source_reference`: 출처 또는 사용자 입력 표시

점수와 근거는 결과의 `provenance`에 그대로 남는다. 후보별 기준시각이 30일을 초과해 다르면 경고한다.

## 4. 정규화와 점수 계산

비교 가능한 후보 집합 안에서 기준별 min-max 정규화를 적용한다.

```text
낮을수록 좋은 기준 = (최댓값 - 후보값) / (최댓값 - 최솟값) × 100
높을수록 좋은 기준 = (후보값 - 최솟값) / (최댓값 - 최솟값) × 100
기여도 = 정규화 점수 × 가중치 / 100
총점 = 모든 기여도의 합
```

모든 후보 값이 같으면 해당 기준은 차이를 만들지 못하므로 정규화 점수를 50으로 둔다. 금액과 점수 계산은 `Decimal`을 사용하고 화면용 총점·기여도는 소수 둘째 자리까지 반올림한다.

가중치가 0보다 큰 기준이 하나라도 누락되면 해당 후보를 `not_comparable`로 표시하고 순위에서 제외한다. 결측값에 평균, 0 또는 최악값을 임의로 넣지 않는다.

## 5. 동률과 압도적 열위

- 총점 차이가 0.01점 이하면 같은 순위로 처리한다.
- 동률 후보는 `candidate_id`로만 결정적으로 정렬하며 순위 자체는 같다.
- 한 후보가 모든 사용 기준에서 다른 후보보다 같거나 좋고 하나 이상에서 더 좋으면 후자를 Pareto 열위로 표시한다.
- Pareto 열위 후보는 삭제하지 않고 `PARETO_DOMINATED`와 `dominated_by`를 제공한다.
- 하드 제약 통과 후보가 하나뿐이면 `partial`과 `ONLY_FEASIBLE_CANDIDATE`를 반환한다.

## 6. 민감도 분석

기본적으로 각 사용 기준의 가중치를 각각 ±10%p 조정한다. 대상 기준 외의 가중치는 기존 비율대로 재배분하여 합을 100으로 유지한다.

각 시나리오는 다음을 반환한다.

- 변경 기준과 정확한 증감 폭
- 조정된 전체 가중치
- 시나리오별 공동 1위 후보
- 후보별 순위
- 후보별 최선·최악 순위
- 가중치 변경 시 1위가 바뀌는지 여부

따라서 “비용 비중을 40%에서 50%로 올리면 A가 1위”처럼 순위가 바뀌는 조건을 숫자로 표시할 수 있다.

## 7. 트레이드오프와 설명 분리

각 후보를 결정적으로 선택한 기준 1위 후보와 비교하여 더 좋은 기준과 불리한 기준을 구조화된 목록으로 반환한다. 점수·순위·근거는 코드가 만들고, 자연어 추천 문장은 Phase 8~9에서 이 구조화 결과만 설명한다. 이후 자연어의 숫자와 후보 ID를 원본 계산 결과와 별도로 검증할 수 있다.

## 8. API

```http
POST /rankings/compare
```

응답 주요 필드:

- `status`: `ranked`, `partial`, `not_comparable`
- `ranking_version`, `input_sha256`
- 후보별 `rank`, `total_score`, `contributions`
- 하드 제약 실패와 누락 필드
- `dominated_by`, `tradeoffs`
- `sensitivity`
- 비용·통근·위험·인프라·정책의 `provenance`
- 기준시각과 계산 버전 경고

## 9. 구현 파일

- 순위 엔진: `apps/api/homefit_api/ranking.py`
- API: `apps/api/homefit_api/ranking_api.py`
- 테스트: `apps/api/tests/test_ranking.py`
- DB 마이그레이션: `apps/api/migrations/003_phase5_rankings.sql`

DB는 실행별 입력 해시, 가중치, 하드 제약, 비용 시나리오, 경고, 트레이드오프와 민감도를 저장한다. 후보 결과에는 기여도, 결측, 우열 관계와 출처 묶음을 저장하여 후속 설명을 재검증할 수 있다.
