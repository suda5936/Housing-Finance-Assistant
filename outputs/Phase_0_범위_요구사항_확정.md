# 집결정 AI Phase 0 — 범위·요구사항 확정

## 1. Phase 0 결과

- 상태: **완료**
- 기준일: 2026-07-21
- 핵심 모델: **Qwen3-4B (`qwen3:4b`)**
- 실행 방식: **Ollama 로컬 실행**
- 외부 유료 LLM API: 사용하지 않음
- 핵심 사용자: 서울에서 첫 독립을 준비하며 월세·보증부 월세 후보를 비교하는 만 19~39세 청년
- MVP 입력: 사용자 직접 입력을 기본 경로로 하고 이미지·PDF 추출은 보조 경로로 제공
- MVP 출력: 후보 2~3개의 실질 주거비, 정책 사전자격 상태, 비교 순위, 근거, 계약 전 체크리스트

Qwen3-4B는 계산이나 정책 자격을 결정하지 않는다. 모델은 질문 분류, 누락 정보 질문, 검색된 근거 요약과 결정 결과 설명만 담당한다. 금액 계산은 계산 엔진, 자격 판정은 버전형 규칙 엔진이 담당한다.

## 2. 제품 요구사항(PRD)

### 2.1 해결할 문제

사용자가 월세만 보고 집을 선택하지 않도록 관리비, 보증금 조달비용, 초기비용, 통근비와 지원제도를 함께 계산하고 비교한다. 여러 정책 사이트의 내용을 단순히 나열하지 않고 사용자 조건에 필요한 추가 정보를 질문한 뒤, 검증 가능한 도구의 결과를 하나의 의사결정 카드로 조합한다.

### 2.2 MVP 목표

가상의 서울 청년 사용자가 후보 주택 3개를 등록한 뒤 5분 안에 다음 결과를 얻는 데모를 완성한다.

1. 후보별 월평균 실질 주거비와 계약기간 총비용
2. 월 세후소득 대비 주거비 부담률
3. 선정 정책 3개의 사전자격 상태와 공식 출처
4. 비용·통근·보증금·면적·주의사항에 따른 후보 순위
5. 누락 정보, 공식 확인사항과 다음 행동
6. 월세·관리비·금리·출근일수 변경 시 재계산 결과

### 2.3 MVP 포함 범위

- 서울 소재 월세·보증부 월세 후보 2~3개
- 익명 데모 세션
- 사용자 직접 입력과 수정 가능한 OCR 보조 입력
- 개인의 월 세후소득, 유동자산, 가구형태, 무주택 여부와 통근 조건
- 정책 3개의 버전형 규칙 판정
- 결정론적 주거비 계산과 가중합 기반 후보 비교
- 공식 출처, 확인일, 계산 가정과 불확실성 표시
- 입력값을 변경한 재계산
- 원본 문서 즉시 삭제 및 자동 삭제

### 2.4 MVP 제외 범위

- 로그인·소셜 로그인과 장기 사용자 계정
- 전국 지자체 정책 실시간 수집
- 실제 대출 승인 가능성 또는 최종 수혜 확정
- 계약의 법적 효력과 전세사기 여부 판정
- 매물 플랫폼 자동 수집
- 은행·지도 유료 API 연결
- LLM 파인튜닝
- 음성 인터페이스와 모바일 앱

## 3. Qwen3-4B 적용 결정

### 3.1 실행 기준

| 항목 | 결정 |
| --- | --- |
| 모델 | `qwen3:4b` |
| 런타임 | 로컬 Ollama |
| API 형식 | Ollama HTTP API를 내부 LLM 어댑터로 감쌈 |
| 기본 컨텍스트 예산 | 8,192 tokens |
| 기본 최대 출력 | 1,024 tokens |
| temperature | 구조화 작업 `0.0`, 설명 작업 `0.2` |
| 출력 | Pydantic/JSON Schema 검증을 통과한 JSON 또는 실패 |
| 재시도 | 형식 오류 시 1회만 수정 재시도, 이후 수동 경로 |
| 개인정보 | 외부 LLM 전송 없음, 원문 전체를 프롬프트 로그에 저장하지 않음 |

컨텍스트와 출력 수치는 제품의 초기 운영값이며 모델 자체 한도를 뜻하지 않는다. Phase 11 평가 결과에 따라 변경하되 설정값과 실행 기록에 버전을 남긴다.

### 3.2 모델이 담당하는 작업

- 사용자 발화의 의도와 입력 필드 후보 추출
- 이미 OCR된 짧은 텍스트에서 필드 후보를 JSON으로 구조화
- 누락 필드 중 다음에 물어볼 질문 1개 작성
- 규칙·계산·검색 결과를 쉬운 한국어로 설명
- 근거가 제공된 범위 안에서 트레이드오프 요약

### 3.3 모델이 담당하지 않는 작업

- 정책 적격·부적격 최종 판정
- 금액, 이자, 부담률 계산
- 출처가 없는 정책 조건 생성
- OCR 없이 스캔 PDF 전체를 직접 해석
- 자유로운 웹 검색과 임의 도구 실행
- 계약 안전성·법적 효력·대출 승인 단정

### 3.4 4B 모델 보완 전략

1. 긴 공고문은 작은 청크로 검색하고 관련 근거만 모델에 제공한다.
2. 한 번의 호출에 하나의 작업만 요청한다.
3. 에이전트는 자유 계획형이 아니라 코드로 정의한 상태 머신으로 구성한다.
4. 모든 모델 출력은 스키마, 허용값, 출처 ID와 숫자 대조 검증을 거친다.
5. 도구 인자는 모델이 만든 값을 바로 사용하지 않고 사용자 확정값과 서버 검증값으로 구성한다.
6. 모델 장애·타임아웃·검증 실패 시 수동 입력과 템플릿 설명으로 계속 진행한다.

## 4. 정책 선정

MVP 정책은 서울 청년 월세 사용자와 직접 연결되고 공식 원문을 확보할 수 있는 다음 3개로 제한한다.

| ID | 정책 | 역할 | Phase 0 결정 |
| --- | --- | --- | --- |
| `POL-SEOUL-RENT-2026` | 2026 서울시 청년월세지원 | 월 임차료 지원 사전자격 확인 | 2026 공고 스냅샷 사용 |
| `POL-NHUF-YOUTH-DEPOSIT-MONTHLY` | 청년전용 보증부월세대출 | 보증금·월세 조달 선택지 확인 | 분석 실행일 기준 공식 상품 페이지 버전 사용 |
| `POL-NHUF-HOUSING-STABILITY-MONTHLY` | 주거안정월세대출 | 월세 대출 선택지 확인 | 분석 실행일 기준 공식 상품 페이지 버전 사용 |

정책 결과는 `신청 가능`이 아니라 **사전자격 상태**로 표현한다. 서울시 청년월세지원처럼 자격을 충족해도 정원 초과 시 추첨될 수 있는 정책은 `ELIGIBLE` 대신 사용자 화면에 `기본요건 충족 가능 — 선정 보장 아님`으로 설명한다.

공식 출처 레지스트리의 최초 기준은 다음과 같다.

- 서울주거포털 2026 청년월세지원 사업개요 및 공고문
- 주택도시기금 청년전용 보증부월세대출 상품안내
- 주택도시기금 주거안정월세대출 상품안내

Phase 4에서 원문 파일, 해시, 공고일, 시행기간, 확인일과 규칙 검토자를 등록하기 전에는 실제 자격 결과를 서비스에 노출하지 않는다.

## 5. 용어집과 계산 가정

| 용어 | MVP 정의 |
| --- | --- |
| 기준소득 | 사용자가 입력한 월 세후소득 |
| 자산 | 정책 판정용 자산과 실제 사용 가능한 보증금 자금을 별도 입력 |
| 보증부 월세 | 임차보증금과 월세를 함께 지급하는 임대차 후보 |
| 관리비 | 월 고정 관리비. 미상일 때 기준값을 임의 생성하지 않음 |
| 관리비 미상 | 기준 계산에서 제외하지 않고 `정보부족`으로 표시하고 보수 시나리오 입력을 요청 |
| 자기자금 기회비용 | MVP 기본값 0원, 선택 시 사용자 입력 연이율 적용 |
| 차입 보증금 비용 | 차입원금 × 연이율 ÷ 12 |
| 초기비용 | 중개보수·이사비 등 반환되지 않는 일회성 비용 |
| 초기비용 환산기간 | 계약기간 개월 수 |
| 월 통근비 | 사용자가 입력한 왕복 1회 비용 × 월 출근일수 |
| 실질 주거비 | 월세 + 관리비 + 보증금 월 비용 + 월 통근비 + 초기비용 월 환산액 - 확정 지원금 |
| 예상 지원금 | 사전자격만 충족한 지원금. 순위 기본 계산에서는 차감하지 않고 별도 시나리오로 표시 |
| 주거비 부담률 | 월평균 실질 주거비 ÷ 월 세후소득 × 100 |
| 확인 필요 | 공식기관·은행 또는 원문 공고에서 추가 확인해야 하는 상태 |

예상 지원금은 실제 선정·승인 전까지 기본 비용에서 차감하지 않는다. 이 원칙으로 지원을 받지 못했을 때 지나치게 낙관적인 추천이 생성되는 것을 막는다.

## 6. 결과 상태

### 정책 사전자격

- `ELIGIBLE`: 입력 기준상 기본요건 충족 가능
- `INELIGIBLE`: 명시 조건 미충족
- `MISSING_INFORMATION`: 필요한 입력 부족
- `OFFICIAL_CHECK_NEEDED`: 예외·중복수혜·최신 조건 확인 필요
- `EXPIRED`: 선택한 정책 버전의 신청기간 종료

### 후보 비교

- `RECOMMENDED`: 필수 입력이 충분하고 1순위로 계산됨
- `CONDITIONAL`: 누락·지원 미확정·민감도에 따라 순위가 바뀔 수 있음
- `NOT_RECOMMENDED`: 예산 초과 또는 다른 후보보다 명확히 불리함
- `NOT_COMPARABLE`: 핵심 비용 입력이 부족하여 비교 불가

## 9. 대표 시나리오

### 정상 시나리오 3건

1. 후보 3개와 모든 필수값을 입력하면 비용·정책·순위·근거가 포함된 카드를 생성한다.
2. 매물 이미지 OCR 값을 사용자가 수정·확정한 뒤 동일한 계산 경로를 완료한다.
3. 출근일수를 변경하면 통근비, 총비용과 후보 순위가 재계산된다.

### 경계 시나리오 5건

1. 정책 연령 상한일과 생일 경계에서 기준일을 적용한다.
2. 소득·보증금·월세가 정책 기준과 정확히 같은 값을 처리한다.
3. 월세가 기준을 넘지만 보증금 환산액 합산 조건을 적용해야 한다.
4. 관리비가 범위로만 제시되면 기준·보수 시나리오를 분리한다.
5. 후보 총점 차이가 2점 미만이면 1위 단정 대신 민감한 추천으로 표시한다.

### 실패 시나리오 6건

1. Ollama가 꺼져 있으면 템플릿 질문·설명과 수동 입력 경로를 제공한다.
2. Qwen 출력이 JSON Schema를 두 번 통과하지 못하면 `MODEL_OUTPUT_INVALID`로 종료한다.
3. OCR이 실패하면 원본을 재시도하지 않고 수동 입력 화면으로 이동한다.
4. 정책 출처 또는 시행기간이 없으면 해당 정책 판정을 비활성화한다.
5. 세후소득이 0 이하이거나 금액이 음수이면 계산을 거부하고 필드를 지목한다.
6. 문서에 주민등록번호 형태가 감지되면 마스킹 전까지 LLM 입력과 로그 기록을 차단한다.

## 10. 데이터 사용·보관·삭제 정책 초안

- 익명 세션 ID만 사용한다.
- 정확한 주소는 계산에 필요할 때만 사용하고 저장 시 동 단위 또는 좌표 정밀도를 낮춘다.
- 업로드 원본은 로컬 개발 저장소 밖의 `.local-data/uploads`에 저장한다.
- 원본 기본 보관기간은 24시간이며 사용자가 즉시 삭제할 수 있다.
- 주민등록번호, 계좌번호, 신분증 원본은 입력 금지 대상으로 처리한다.
- 프롬프트 로그에는 원문 전체와 민감값을 저장하지 않는다.
- 삭제 시 원본, 파생 OCR 텍스트, 추출값, 세션 데이터와 캐시를 함께 삭제한다.
- 대회 평가용 데이터는 실제 개인정보가 없는 합성 데이터만 사용한다.

## 13. Phase 1 전달사항

1. Ollama 어댑터와 `qwen3:4b` 상태 확인 API를 만든다.
2. 모델이 없어도 수동 입력·계산 데모가 가능하도록 기능 플래그를 둔다.
3. 모델 요청·응답 DTO를 먼저 정의하고 JSON Schema 검증을 적용한다.
4. 자유 계획형 에이전트 프레임워크는 도입하지 않는다.
5. 정책·계산 도구는 LLM과 별도 Python 모듈로 유지한다.
6. 모델 컨텍스트·출력·타임아웃·재시도 값을 환경변수로 관리한다.
7. 실제 PC에서 Qwen3-4B 응답속도를 측정해 `NFR-PERF-01` 기준을 보정한다.
