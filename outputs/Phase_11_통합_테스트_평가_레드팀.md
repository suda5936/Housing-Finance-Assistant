# Phase 11 통합 테스트·평가·레드팀

## 1. 완료 범위

Phase 11에서는 기존 단위·통합 테스트를 버전 고정 골든 세트와 하나의 출시 평가 하네스로 묶었다.

```text
합성 골든 데이터 로드
→ 비용 수기 정답 비교
→ 정책 정상·경계·누락·만료·미검토 상태 비교
→ 순위·점수·동률 비교
→ 정책 검색 Hit@K·인용 정합성 평가
→ 정상 문서·인젝션 문서 분류
→ 자동 출시 차단 항목 집계
→ 모바일·정책 검토 수동 게이트 병합
```

상태는 **자동 평가 완료, 수동 출시 게이트 2건 대기**다. 자동 차단 이슈는 0건이지만 실제 모바일 확인과 정책 독립 검토 증거가 없으므로 `release_ready=false`를 유지한다.

## 2. 산출물

| 파일 | 역할 |
|---|---|
| `apps/api/homefit_api/evaluation_data/release_cases.json` | 버전 고정 합성 골든·경계·공격 데이터 |
| `apps/api/homefit_api/evaluation.py` | 영역별 평가와 출시 게이트 집계 |
| `apps/api/tests/test_evaluation.py` | 재현성, 골든 변조 탐지, 수동 게이트 분리 테스트 |
| `package.json`의 `eval:release` | 팀원이 동일 평가를 실행하는 명령 |

## 3. 평가 데이터셋

데이터셋 버전은 `phase11-release-v1`, 기준일은 2026-07-25다. 실제 개인정보나 실제 계약서를 포함하지 않는다.

### 3.1 비용 골든 세트

수기로 검산한 합성 후보의 기준 시나리오를 정확히 비교한다.

| 항목 | 기대값 |
|---|---:|
| 월평균 실질 주거비 | 608,667원 |
| 계약기간 총비용 | 7,304,000원 |
| 소득 대비 부담률 | 24.35% |

문자열 단위까지 정확히 비교해 반올림이나 통화 단위가 바뀌면 자동 출시 차단 이슈가 된다.

### 3.2 정책 골든·경계 세트

- 정상 기본요건 충족 → `ELIGIBLE`
- 중위소득 48% 배타 하한 → `INELIGIBLE`
- 중위소득 150% 포함 상한·39세 포함 상한 → `ELIGIBLE`
- 보증금 누락 → `MISSING_INFORMATION`
- 신청기간 종료 → `EXPIRED`
- 실제 기본 정책의 검토 미완료 → `OFFICIAL_CHECK_NEEDED`

`approved_fixture=true` 사례는 규칙 경계값 자체를 시험하기 위한 합성 승인 사본이다. 실제 정책 레지스트리를 변경하거나 승인 상태로 만들지 않는다.

### 3.3 순위 골든 세트

- 기본 후보 3개의 순위 `B → A → C`
- 점수 `65.00 → 58.33 → 27.50`
- 완전히 같은 후보 2개의 dense rank `[1, 1]`
- 동일 입력의 동일 순위·점수 재현

### 3.4 검색·인용 세트

정책별 합성 질문 3개로 다음을 각각 평가한다.

- 기대 공식 문서가 상위 K개 안에 존재하는지
- 기대 조건 코드와 인용이 직접 연결되는지

### 3.5 문서 공격 세트

- 정상 보증금·월세 문장은 인젝션으로 오탐하지 않음
- 한국어 `이전 지시를 무시` 공격 탐지
- 영어 `Ignore previous instructions`와 전체 도구 호출 요구 탐지

## 4. 평가 하네스

`evaluate_release()`는 모든 검사 결과를 `ReleaseEvaluationReport`로 반환한다.

- `checks`: 기대값, 관측값, 통과 여부와 차단 여부
- `metrics`: 영역별 통과 수, 전체 수와 정확도
- `automated_blockers`: 실패한 자동 차단 검사 ID
- `pending_manual_gates`: 증거가 필요한 사람 검토
- `automated_gate_passed`: 자동 검사만 통과했는지
- `release_ready`: 자동 검사와 수동 게이트가 모두 끝났는지

골든 기대값 하나를 임의로 `0`으로 바꾸는 회귀 테스트에서 해당 검사 ID가 즉시 `automated_blockers`에 포함됨을 검증했다. 평가 결과를 두 번 실행했을 때 객체 전체가 동일해 시간이나 외부 모델에 따라 변하지 않는다.

## 5. 자동 평가 결과

| 영역 | 통과/전체 | 정확도 |
|---|---:|---:|
| 비용 | 4/4 | 100% |
| 정책 | 6/6 | 100% |
| 순위 | 5/5 | 100% |
| 검색·인용 | 2/2 | 100% |
| 문서 안전 | 3/3 | 100% |
| 합계 | 20/20 | 100% |

- 자동 출시 차단 이슈: **0건**
- 자동 게이트: **통과**
- 대기 중 수동 게이트: **2건**

100%는 이 작은 합성 골든 세트에 대한 결과이며 실사용 OCR 정확도나 사용자 이해도 100%를 뜻하지 않는다.

## 6. 필수 흐름·장애 복구 검증

기존 자동 테스트와 새 평가를 함께 실행해 다음 수직 흐름을 확인한다.

1. 동의와 익명 세션 생성
2. 프로필과 후보 2개 저장
3. 문서 없음 또는 OCR 실패 시 수동 입력
4. 정책 판정과 근거 검색
5. 후보별 비용 계산
6. 다기준 순위와 기여도 계산
7. 검증 게이트
8. 정책 검토 전 `official_check` 종료
9. 실행 재조회·재개
10. 세션 삭제 후 원문·캐시 접근 차단

장애 주입 테스트는 OCR 실행 불가, LLM 중단·모델 누락·원격 URL, 알 수 없는 정책, 도구 실패, 호출·단계 한도, 비용 provenance 변조와 순위 결측을 다룬다. 어떤 경우에도 검증되지 않은 의사결정 카드를 성공 결과로 반환하지 않는다.

| 관리비 누락·범위 | 누락 보류 또는 3시나리오 계산 | 통과 | 비용 테스트 |

Phase 11에서 발견한 구현 결함은 없었다. 대신 이전 단계의 위험을 숨기지 않도록 수동 확인 2건을 `release_ready` 계산에 새로 포함했다.

## 8. 알려진 한계

- 실제 촬영 이미지의 저화질·회전·손글씨 OCR 정확도는 측정하지 않았다.
- 평가 데이터는 합성 사례이며 모집단 성능을 대표하지 않는다.
- 정책 3개는 실제 독립 검토 전이라 서비스가 계속 `official_check`를 반환한다.
- 실제 모바일 화면 완료시간, 이해도와 오류 복구율은 사용자 테스트가 필요하다.
- 지도 API 없이 통근시간과 교통비를 사용자가 직접 입력한다.
- 생활 인프라 가중치는 신뢰할 출처가 없으면 0으로 유지한다.
- 운영 부하·다중 인스턴스·중앙 로그·알림은 Phase 12 배포 환경에서 측정한다.

이 한계는 의사결정 카드의 공식확인 상태·계산 가정·서비스 한계와 Phase 9·10 운영 문서에 반영돼 있다.

## 9. 재현 방법

프로젝트 루트에서 다음 한 줄을 실행한다.

```powershell
npm run eval:release
```

이 명령은 프로젝트 `.venv` Python과 저장소의 고정 JSON 데이터만 사용한다. Ollama, 유료 API, 네트워크, PostgreSQL이 필요하지 않다. 자동 차단 검사가 실패하면 종료 코드 1을 반환한다. 수동 게이트 대기는 보고서의 `release_ready=false`로 표시하지만 자동 평가 명령 자체는 성공한다.

전체 테스트:

```powershell
.\.venv\Scripts\python.exe -m pytest apps/api/tests
npm run test:web
```

## 13. Phase 12 전달사항

- 수동 게이트 2건을 완료하고 증거 위치를 평가 데이터와 배포 체크리스트에 연결한다.
- 합성 데모 3세트와 문서 파일을 신규 환경에서 재현한다.
- 스테이징 TLS·암호화·악성코드 검사·중앙 로그와 지표를 연결한다.
- 부하·지연·장애 복구 측정값을 `ReleaseEvaluationReport`의 운영 평가에 추가한다.
- 3분·5분·10분 시연 시나리오에 수동 폴백과 `official_check` 설명을 포함한다.
